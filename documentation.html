1. Introduction et objectifs

Le Reverse Proxy Manager est une solution auto-hebergee de gestion dynamique de reverse proxy Nginx via interface web. L'application fonctionne dans un conteneur Docker unique.

Objectifs

  • Simplifier la configuration de reverse proxy sans editer les fichiers Nginx.
  • Presenter les services disponibles sous forme de tuiles cliquables sur une page d'accueil publique.
  • Securiser l'administration par mot de passe avec protection anti-brute-force et historique des blocages.
  • Automatiser la validation et le rechargement de Nginx apres chaque modification.
  • Persister toutes les donnees (routes, tuiles, logos, historique) dans un volume Docker.

5. Configuration Docker

5.1 Dockerfile

FROM nginx:1.27-alpine
RUN apk add --no-cache python3 py3-pip supervisor
WORKDIR /app
COPY app/requirements.txt .
RUN pip3 install --no-cache-dir --break-system-packages -r requirements.txt
COPY app/ /app/
COPY nginx/nginx.conf /etc/nginx/nginx.conf
RUN mkdir -p /etc/nginx/conf.d /data /etc/nginx/certs
RUN echo "# No routes configured" > /etc/nginx/conf.d/proxy.conf
COPY supervisord.conf /etc/supervisord.conf
EXPOSE 80 443
CMD ["supervisord", "-c", "/etc/supervisord.conf"]

5.2 Docker Compose

services:
  reverseproxy:

6. Configuration Nginx

6.1 Configuration principale

Le fichier nginx.conf definit trois blocs serveur :

BlocPortRoleserver_name
Administration + Accueil443 (SSL, default)Proxy vers Flask :5000_ (catch-all)
Redirection HTTP80301 HTTP → HTTPS_
Routes dynamiques443+ (SSL)Proxy vers backendsDomaine configure

En-tetes proxy transmis

En-teteValeurDescription
Host$hostNom d'hote original
X-Real-IP$remote_addrIP reelle du client
X-Forwarded-For$proxy_add_x_forwarded_forChaine de proxys
X-Forwarded-Proto$schemeProtocole original

6.2 Generation dynamique des routes

Chaque route genere un bloc server {} dans /etc/nginx/conf.d/proxy.conf avec support WebSocket natif (Upgrade/Connection). Si le port d'ecoute est 80, les directives SSL sont automatiquement omises.

8.3 Historique des blocages

Chaque blocage d'IP est enregistre dans /data/block_history.json, un fichier persiste sur le volume Docker. L'historique survit aux redemarrages du conteneur.

ProprieteValeur
Fichier de stockage/data/block_history.json
Capacite200 derniers evenements (FIFO)
DeclenchementAutomatique a chaque blocage d'IP (6e echec)
PersistanceVolume Docker proxy-data, survit aux redemarrages
PurgeBouton "Effacer l'historique" dans l'admin

12. Guide d'utilisation

Gerer les tuiles

  1. Connectez-vous sur /admin.
  2. Section "Gestion des tuiles" : remplir nom, URL, logo (optionnel), position.
  3. Cliquer Ajouter. La tuile apparait immediatement sur /.
  4. Modifier : edition en ligne (nom, URL, position). Supprimer : avec confirmation.

Gerer les routes proxy

  1. Section "Routes de reverse proxy" : remplir domaine, cible, port.
  2. Cliquer Ajouter. Nginx est valide (nginx -t) et recharge automatiquement.

Gerer la securite (IP)

  1. IP bloquees (section rouge) : voir les IP actuellement bloquees, temps restant, bouton Debloquer.
  2. Historique (section orange) : consulter tous les blocages passes avec date, IP et raison.
  3. Bouton Effacer l'historique pour purger le journal (avec confirmation).
Rappel : Les tuiles et les routes proxy sont independantes. Une tuile peut pointer vers n'importe quelle URL.
