Reverse Proxy Manager

Interface web d'administration pour gerer un reverse proxy Nginx, avec authentification LDAP, gestion des certificats SSL et tableau de bord personnalisable. v2.2

Introduction

Reverse Proxy Manager est une application web autonome permettant de configurer et gerer un reverse proxy Nginx via une interface graphique moderne. L'application est conteneurisee avec Docker et offre une solution cle en main pour le routage de services internes.

Reverse Proxy

Gestion CRUD des routes Nginx avec rechargement automatique, support WebSocket et HTTPS.

Page d'accueil

Tuiles de services personnalisables avec logos, organisees en sections.

Authentification

Connexion locale ou LDAP/SSO avec verification de groupe d'administration.

Certificats SSL

Generation de certificats auto-signes, import de certificats personnalises, configuration Let's Encrypt.

Securite

Rate limiting, blocage automatique d'IP apres tentatives echouees, historique des blocages.

Sauvegarde

Export/import complet de la configuration au format JSON, incluant les logos.

Architecture

L'application fonctionne dans un conteneur Docker unique contenant trois services geres par Supervisor :

+-------------------------------------------------------------+ | Conteneur Docker | | | | +-------------------+ +---------------------------+ | | | Nginx | | Flask / Gunicorn | | | | (port 80/443) |---->| (port 5000 interne) | | | | | | | | | | - SSL termination | | - Interface admin | | | | - Reverse proxy | | - API de gestion | | | | - Conf dynamique | | - Auth locale + LDAP | | | +-------------------+ +---------------------------+ | | | | | | v v | | /etc/nginx/conf.d/ /data/ (volume persistant) | | proxy.conf - routes.json | | (genere par Flask) - tiles.json | | - ldap_config.json | | +-------------------+ - block_history.json | | | Supervisor | - certs/ | | | (gestionnaire) | - logos/ | | +-------------------+ | +-------------------------------------------------------------+ | Port 888 (host) --> 443 (conteneur)

Flux de fonctionnement

  1. Supervisor demarre Gunicorn (Flask) en priorite, puis Nginx.
  2. Nginx ecoute sur les ports 80 (redirection HTTP) et 443 (HTTPS).
  3. Les requetes vers le domaine d'admin sont proxyfiees vers Flask sur le port 5000 interne.
  4. Les requetes vers les domaines configures sont proxyfiees vers les services cibles.
  5. Flask genere dynamiquement le fichier /etc/nginx/conf.d/proxy.conf et recharge Nginx.

Prerequis

  • Docker ≥ 20.10
  • Docker Compose ≥ 2.0
  • Un systeme Linux (teste sur Ubuntu/Debian)
  • Port 888 disponible sur la machine hote (configurable)

Installation

1. Cloner le depot

git clone https://github.com/ttheoha/IA_reversproxy.git
cd IA_reversproxy

2. Configurer les variables d'environnement

Editez le fichier docker-compose.yml pour adapter les parametres :

services:
  reverseproxy:
    build: .
    container_name: reverseproxy_th
    ports:
      - "888:443"        # Port d'acces (modifiable)
    volumes:
      - proxy-data:/data
    environment:
      - SECRET_KEY=votre-cle-secrete-ici    # Cle de session Flask
      - ADMIN_PASSWORD=votre-mot-de-passe   # Mot de passe admin local

volumes:
  proxy-data:
Important : Changez imperativement SECRET_KEY et ADMIN_PASSWORD avant tout deploiement en production.

3. Construire et demarrer

# Construction et demarrage
docker compose up -d --build

# Verifier que le conteneur fonctionne
docker ps | grep reverseproxy_th

# Consulter les logs
docker logs -f reverseproxy_th

4. Acceder a l'interface

Ouvrez votre navigateur et accedez a :

https://<IP-DE-VOTRE-SERVEUR>:888
Un certificat auto-signe est genere automatiquement au premier demarrage. Votre navigateur affichera un avertissement de securite que vous pouvez accepter.

Premier lancement

Lors du premier acces, suivez ces etapes :

1 Connexion — Rendez-vous sur /login et connectez-vous avec le mot de passe defini dans ADMIN_PASSWORD.

2 Certificat SSL — Allez dans Certificats pour generer un certificat auto-signe personnalise ou importer votre propre certificat.

3 Routes proxy — Configurez vos premieres routes de reverse proxy dans le tableau de bord d'administration.

4 Tuiles — Creez des sections et ajoutez des tuiles pour organiser vos services sur la page d'accueil.

5 LDAP (optionnel) — Si vous utilisez un annuaire LDAP, configurez l'integration dans la section LDAP / SSO.

6 Sauvegarde — Effectuez une premiere sauvegarde pour securiser votre configuration.

Routes de reverse proxy

Les routes de reverse proxy permettent de rediriger les requetes entrantes vers vos services internes en fonction du nom de domaine (Host header).

Ajouter une route

Renseignez les champs suivants dans le formulaire du tableau de bord :

ChampDescriptionExemple
Domaine / HostLe nom de domaine ou hostname ciblecloud.exemple.com
CibleURL ou IP:port du service backendhttp://192.168.1.10:8080
PortPort d'ecoute Nginx (443 par defaut pour HTTPS)443

Configuration Nginx generee

Pour chaque route, un bloc server Nginx est automatiquement genere :

server {
    listen 443 ssl;
    server_name cloud.exemple.com;

    ssl_certificate     /data/certs/selfsigned.crt;
    ssl_certificate_key /data/certs/selfsigned.key;

    location / {
        proxy_pass http://192.168.1.10:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}

Fonctionnalites des routes

  • Ajout — Creation d'une nouvelle route via le formulaire
  • Modification — Edition inline (domaine, cible, port) via le bouton "Modifier"
  • Suppression — Suppression avec confirmation via le bouton "Supprimer"
  • Support WebSocket — Les headers Upgrade et Connection sont automatiquement configures
  • Rechargement automatique — Nginx est recharge automatiquement apres chaque modification
  • Validation — La configuration est validee avant le rechargement (nginx -t)
Si le port est 80, le bloc sera configure sans SSL. Pour tous les autres ports, SSL est active avec les certificats situes dans /data/certs/.

Certificats SSL/TLS v2.2

La page de gestion des certificats (/admin/certs) permet de configurer le chiffrement HTTPS pour le reverse proxy.

Certificat actuel

L'onglet "Certificat actuel" affiche les informations du certificat en place :

  • Sujet (CN) — Nom commun du certificat
  • Emetteur — Autorite de certification
  • Valide du / au — Periode de validite
  • Empreinte SHA-256 — Fingerprint du certificat

Certificat auto-signe

Generez un nouveau certificat auto-signe avec les parametres :

  • Nom commun (CN) — Le hostname principal (ex: proxy.exemple.com)
  • Duree de validite — En jours (defaut : 3650 jours / 10 ans)
# Commande executee en interne :
openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
  -keyout /data/certs/selfsigned.key \
  -out /data/certs/selfsigned.crt \
  -subj "/CN=proxy.exemple.com"

Certificat personnalise

Importez votre propre certificat et cle privee :

  • Certificat — Fichier .crt ou .pem
  • Cle privee — Fichier .key ou .pem
Les fichiers sont valides avec openssl avant d'etre enregistres. Un certificat invalide sera rejete avec un message d'erreur.

Let's Encrypt (experimental)

L'interface de configuration Let's Encrypt permet de preparer les parametres pour une integration future :

  • Domaine — Le domaine a certifier
  • Email — Email de contact pour les notifications
  • Renouvellement automatique — Option activable
La generation automatique de certificats Let's Encrypt n'est pas encore implementee. La configuration est sauvegardee pour une integration dans une version future.

LDAP / SSO v2.2

L'integration LDAP permet d'authentifier les administrateurs via un annuaire d'entreprise (Active Directory, OpenLDAP, etc.).

Configuration

ParametreDescriptionExemple
Activer LDAPActive/desactive l'authentification LDAPOui / Non
ServeurAdresse du serveur LDAPldap.exemple.com
PortPort du serveur (389 pour LDAP, 636 pour LDAPS)389
SSL/TLSActiver la connexion chiffreeOui / Non
Bind DNDN du compte de service pour la recherchecn=admin,dc=exemple,dc=com
Bind PasswordMot de passe du compte de service-
Base DNBase de recherche des utilisateursou=users,dc=exemple,dc=com
Filtre utilisateurFiltre LDAP avec {username} comme variable(uid={username})
Groupe adminDN du groupe d'administrateurs (optionnel)cn=admins,ou=groups,dc=exemple,dc=com

Flux d'authentification LDAP

  1. L'utilisateur saisit son identifiant LDAP et son mot de passe
  2. L'application se connecte au serveur LDAP avec le compte de service (Bind DN)
  3. Une recherche est effectuee dans la Base DN avec le filtre utilisateur
  4. Si l'utilisateur est trouve, une tentative de bind avec ses identifiants est effectuee
  5. Si un groupe admin est configure, l'appartenance de l'utilisateur est verifiee
  6. En cas de succes, la session est creee

Test de connexion

Le bouton "Tester la connexion" execute un test complet en 5 etapes :

  • Connexion au serveur LDAP
  • Bind avec le compte de service
  • Recherche dans la base DN
  • Verification du filtre utilisateur
  • Verification du groupe admin (si configure)

Mode de connexion mixte

Lorsque LDAP est active, la page de connexion affiche un champ "Identifiant LDAP" supplementaire. L'authentification essaie d'abord le LDAP, puis se rabat sur le mot de passe local si le champ identifiant est vide ou si LDAP echoue.

Le mot de passe local (ADMIN_PASSWORD) reste toujours fonctionnel comme solution de secours, meme lorsque LDAP est active.

Securite & blocage IP

Rate limiting

Le systeme protege l'interface de connexion contre les attaques par force brute :

  • Seuil — 6 tentatives echouees par adresse IP
  • Duree de blocage — 10 minutes apres depassement du seuil
  • Deblocage automatique — L'IP est automatiquement debloquee a l'expiration du delai
  • Reinitialisation — Le compteur est remis a zero apres une connexion reussie

Gestion des IP bloquees

Depuis le tableau de bord d'administration :

  • Visualisation des IP actuellement bloquees avec le nombre de tentatives et le temps restant
  • Deblocage manuel immediat d'une IP

Historique des blocages

  • Les 200 derniers evenements de blocage sont conserves
  • Chaque entree contient : date, adresse IP, raison du blocage
  • L'historique est persistant (stocke dans /data/block_history.json)
  • Possibilite d'effacer l'historique complet

Sauvegarde & restauration

Sauvegarde

Le bouton "Sauvegarder (JSON)" genere un fichier de backup contenant :

  • Toutes les routes de reverse proxy
  • Toutes les sections et tuiles (avec positions)
  • Tous les logos (encodes en base64)
  • La configuration LDAP
  • L'historique des blocages IP

Le fichier est telecharge sous le nom reverseproxy_backup.json.

Restauration

  1. Cliquez sur le champ de selection de fichier
  2. Selectionnez un fichier .json de sauvegarde
  3. Cliquez sur "Restaurer"
  4. Confirmez dans la boite de dialogue
Attention : La restauration ecrase toutes les donnees actuelles (routes, tuiles, logos, configuration LDAP). Effectuez une sauvegarde prealable si necessaire.

Configuration Nginx

La configuration Nginx de base se trouve dans nginx/nginx.conf :

Caracteristiques

  • Worker processes — Auto-detecte le nombre de CPU
  • Protocoles SSL — TLSv1.2 et TLSv1.3
  • Redirection HTTP — Toutes les requetes HTTP (port 80) sont redirigees vers HTTPS
  • Serveur par defaut — Les requetes sans correspondance de domaine sont envoyees a Flask
  • Logs — Format combine avec X-Forwarded-For

Configuration dynamique

Le fichier /etc/nginx/conf.d/proxy.conf est genere automatiquement par Flask a chaque ajout, modification ou suppression de route. Ce fichier contient les blocs server pour chaque route configuree.

Depannage

Problemes courants

Le conteneur ne demarre pas

# Verifier les logs
docker logs reverseproxy_th

# Verifier que le port 888 n'est pas utilise
ss -tlnp | grep 888

# Reconstruire l'image
docker compose down
docker compose up -d --build

Erreur "502 Bad Gateway"

  • Verifiez que le service cible est accessible depuis le conteneur
  • Verifiez que l'adresse et le port cible sont corrects
  • Testez la connectivite : docker exec reverseproxy_th curl -I http://<cible>

La configuration Nginx ne se recharge pas

# Tester la configuration
docker exec reverseproxy_th nginx -t

# Recharger manuellement
docker exec reverseproxy_th nginx -s reload

# Voir la config generee
docker exec reverseproxy_th cat /etc/nginx/conf.d/proxy.conf

LDAP : erreur de connexion

  • Verifiez que le serveur LDAP est accessible depuis le conteneur
  • Verifiez les informations de Bind DN et mot de passe
  • Si LDAPS, verifiez que le port est 636 et que SSL est active
  • Utilisez le bouton "Tester la connexion" pour un diagnostic detaille

IP bloquee par erreur

  • Accedez au tableau de bord depuis une autre IP
  • Cliquez sur "Debloquer" en face de l'IP bloquee
  • Ou attendez l'expiration automatique (10 minutes)
  • En dernier recours : docker restart reverseproxy_th (les blocages en memoire sont reinitialises)

Certificat invalide apres upload

  • Verifiez que le certificat et la cle correspondent : 
    openssl x509 -noout -modulus -in cert.crt | md5sum
openssl rsa -noout -modulus -in cert.key | md5sum
  • Les deux empreintes MD5 doivent etre identiques

Reinitialiser completement

# Supprimer le conteneur et le volume
docker compose down -v

# Reconstruire et redemarrer
docker compose up -d --build
Attention : L'option -v supprime le volume de donnees. Toutes les configurations, routes, tuiles et logos seront perdus. Effectuez une sauvegarde au prealable.
