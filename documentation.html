1. Présentation générale

1.1 Objectif de l'application

Reverse Proxy Manager est une application web autonome permettant de configurer et gérer un reverse proxy Nginx via une interface graphique moderne. Conteneurisée avec Docker, elle offre une solution clé en main pour le routage de services internes.

  • Reverse proxy dynamique : gestion CRUD des routes Nginx avec rechargement automatique, support WebSocket et HTTPS
  • Page d'accueil personnalisable : tuiles de services organisées en sections avec logos personnalisés
  • Certificats SSL/TLS : auto-signés, import personnalisé, Let's Encrypt HTTP-01 et DNS-01 OVH (wildcard)
  • Authentification : mot de passe local + LDAP/SSO avec vérification de groupe d'administration
  • Sécurité : rate limiting, blocage automatique d'IP après tentatives échouées, historique des blocages
  • Gestion des sites Nginx : ajout de configurations manuelles, activation/désactivation par symlink
  • Logs en temps réel : consultation des logs Nginx et Certbot depuis l'interface
  • Sauvegarde / Restauration : export et import complet de la configuration au format JSON (routes, tuiles, logos, LDAP)
  • Renouvellement automatique : renouvellement certbot toutes les 12 heures via Supervisor

4. Routes de reverse proxy

Les routes de reverse proxy redirigent les requêtes entrantes vers vos services internes en fonction du nom de domaine (Host header).

4.1 Ajouter une route

Renseignez les champs suivants dans le formulaire du tableau de bord :

ChampDescriptionExemple
Domaine / HostLe nom de domaine ou hostname ciblecloud.exemple.com
CibleURL ou IP:port du service backendhttp://192.168.1.10:8080
Port d'écoutePort d'écoute Nginx (443 par défaut pour HTTPS)443

Fonctionnalités des routes :

  • Ajout — Création d'une nouvelle route via le formulaire
  • Modification — Édition inline (domaine, cible, port) via le bouton « Modifier »
  • Suppression — Suppression avec confirmation
  • Support WebSocket — Headers Upgrade et Connection automatiquement configurés
  • Rechargement automatique — Nginx est validé (nginx -t) et rechargé après chaque modification

4.2 Configuration Nginx générée

Pour chaque route, un fichier .conf est créé dans /etc/nginx/sites-available/ et linké dans /etc/nginx/sites-enabled/ :

server {
    listen 443 ssl;
    server_name cloud.exemple.com;

    ssl_certificate     /etc/letsencrypt/live/cloud.exemple.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/cloud.exemple.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    location / {
        proxy_pass http://192.168.1.10:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
Résolution des certificats : L'application cherche d'abord un certificat Let's Encrypt pour le domaine, puis pour le domaine de base (wildcard), puis se rabat sur le certificat global auto-signé. Si le port est 80, le bloc est configuré sans SSL.

4.3 Gestion des sites

La page /admin/sites permet de gérer les fichiers de configuration Nginx manuellement :

  • Liste des sites — Tous les fichiers .conf de sites-available avec leur statut (activé/désactivé)
  • Activation / Désactivation — Création ou suppression du symlink dans sites-enabled
  • Ajout manuel — Création d'un fichier .conf personnalisé avec validation automatique
  • Suppression — Suppression du fichier et du symlink associé
  • Rollback — Si la configuration est invalide, le site est automatiquement retiré

6. Certificats SSL/TLS

La page /admin/certs centralise toute la gestion des certificats HTTPS.

6.1 Certificat auto-signé

Générez un nouveau certificat auto-signé avec les paramètres :

ParamètreDescriptionDéfaut
Nom commun (CN)Le hostname principallocalhost
Durée de validitéEn jours3650 (10 ans)
# Commande exécutée en interne :
openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
  -keyout /data/certs/selfsigned.key \
  -out /data/certs/selfsigned.crt \
  -subj "/C=FR/ST=Local/L=Local/O=ReverseProxy/CN=proxy.exemple.com"

6.2 Certificat personnalisé

Importez votre propre certificat et clé privée :

  • Certificat — Fichier .crt ou .pem
  • Clé privée — Fichier .key ou .pem
Les fichiers sont validés avec openssl avant d'être enregistrés. Un certificat ou une clé invalide sera rejeté(e) avec un message d'erreur.

6.3 Let's Encrypt — HTTP-01

Le challenge HTTP-01 est le mode standard pour les domaines non-wildcard :

  1. Renseignez l'email de contact et activez le renouvellement automatique
  2. Ajoutez un domaine avec le type de challenge HTTP-01
  3. Certbot crée un fichier dans /var/www/certbot/.well-known/acme-challenge/
  4. Let's Encrypt vérifie le fichier via le port 80 et délivre le certificat
  5. Nginx est automatiquement reconfiguré pour utiliser le nouveau certificat
Prérequis : Le port 80 du serveur doit être accessible depuis Internet pour que Let's Encrypt puisse valider le domaine. Le DNS du domaine doit pointer vers l'IP du serveur.

6.4 Let's Encrypt — DNS-01 OVH (wildcard)

Le challenge DNS-01 via l'API OVH permet d'obtenir des certificats wildcard (ex: *.theoha.ovh) :

Configuration des identifiants OVH

ParamètreDescription
EndpointPoint d'entrée API OVH (ovh-eu par défaut)
Application KeyClé d'application OVH
Application SecretSecret d'application OVH
Consumer KeyClé consommateur OVH

Les identifiants sont stockés dans /data/ovh-credentials.ini (permissions 600) et utilisés par le plugin certbot-dns-ovh.

Propagation DNS : Le challenge DNS-01 attend 60 secondes la propagation des enregistrements TXT. Le timeout total est de 180 secondes.

6.5 Renouvellement automatique

Le script certbot-renew.sh est exécuté toutes les 12 heures par Supervisor :

  1. Vérifie si le renouvellement automatique est activé dans la configuration
  2. Exécute certbot renew --webroot pour tous les certificats existants
  3. Regénère les configurations Nginx pour pointer vers les bons certificats
  4. Recharge Nginx

Les logs de renouvellement sont écrits dans /var/log/reverseproxy/certbot.log.

Actions manuelles

  • Renouveler un domaine — Bouton « Renouveler » pour forcer le renouvellement d'un domaine spécifique
  • Renouveler tout — Bouton « Renouveler tous » pour forcer le renouvellement global
  • Tester un certificat — Vérifie la validité, l'émetteur (Let's Encrypt) et l'expiration (alerte < 30 jours)
  • Retirer un domaine — Supprime le domaine de la gestion Let's Encrypt (fallback sur le certificat global)

7. Authentification

7.1 Authentification locale

L'authentification par défaut utilise un mot de passe unique défini par la variable ADMIN_PASSWORD. La connexion s'effectue sur la page /login.

7.2 LDAP / SSO

L'intégration LDAP permet d'authentifier les administrateurs via un annuaire d'entreprise (Active Directory, OpenLDAP, etc.).

Paramètres de configuration

ParamètreDescriptionExemple
Activer LDAPActive/désactive l'authentification LDAPOui / Non
ServeurAdresse du serveur LDAPldap.exemple.com
PortPort du serveur (389 LDAP, 636 LDAPS)636
SSL/TLSActiver la connexion chiffréeOui / Non
Bind DNDN du compte de service pour la recherchecn=admin,dc=exemple,dc=com
Bind PasswordMot de passe du compte de service—

Flux d'authentification LDAP

1
L'utilisateur saisit son identifiant LDAP et son mot de passe
2
Connexion au serveur LDAP avec le compte de service (Bind DN)
3
Recherche dans la Base DN avec le filtre utilisateur
4
Tentative de bind avec les identifiants de l'utilisateur
5
Vérification de l'appartenance au groupe admin (si configuré)
6
Création de la session en cas de succès

Test de connexion

Le bouton « Tester la connexion » exécute un test complet en 5 étapes : connexion TCP/TLS, bind service, recherche Base DN, vérification du filtre utilisateur, vérification du groupe admin.

7.3 Mode mixte

Lorsque LDAP est activé, la page de connexion affiche un champ « Identifiant LDAP » supplémentaire. L'authentification essaie d'abord le LDAP, puis se rabat sur le mot de passe local si le champ identifiant est vide.

Fallback garanti : Le mot de passe local (ADMIN_PASSWORD) reste toujours fonctionnel comme solution de secours, même lorsque LDAP est activé ou que le serveur LDAP est injoignable.

8. Sécurité & blocage IP

8.1 Rate limiting

Le système protège l'interface de connexion contre les attaques par force brute :

ParamètreValeurDescription
Seuil de blocage6 tentativesNombre de tentatives échouées avant blocage
Durée de blocage10 minutesDurée pendant laquelle l'IP est bloquée
Déblocage automatiqueOuiL'IP est automatiquement débloquée à expiration
RéinitialisationAprès connexion réussieLe compteur est remis à zéro

Depuis le tableau de bord d'administration :

  • Visualisation des IP actuellement bloquées avec le temps restant
  • Déblocage manuel immédiat d'une IP

8.2 Historique des blocages

  • Les 200 derniers événements de blocage sont conservés
  • Chaque entrée contient : date, adresse IP, raison du blocage
  • L'historique est persistant (stocké dans /data/block_history.json)
  • Possibilité d'effacer l'historique complet depuis le tableau de bord

10. Sauvegarde & restauration

Sauvegarde

Le bouton « Sauvegarder (JSON) » génère un fichier reverseproxy_backup.json contenant :

  • Toutes les routes de reverse proxy
  • Toutes les sections et tuiles (avec positions)
  • Tous les logos (encodés en base64)
  • La configuration LDAP
  • L'historique des blocages IP

Restauration

1
Cliquez sur le champ de sélection de fichier
2
Sélectionnez un fichier .json de sauvegarde
3
Cliquez sur « Restaurer »
4
Confirmez dans la boîte de dialogue
Attention : La restauration écrase toutes les données actuelles (routes, tuiles, logos, configuration LDAP). Effectuez une sauvegarde préalable si nécessaire. Les routes Nginx sont automatiquement regénérées et Nginx est rechargé.

C. Configuration Nginx

Le fichier nginx/nginx.conf configure :

  • Worker processes — Auto-détection du nombre de CPU
  • Protocoles SSL — TLSv1.2 et TLSv1.3
  • Client max body size — 1 Go (pour les uploads de logos et backups)
  • Port 80 — Redirection HTTP → HTTPS + support ACME challenge Let's Encrypt
  • Sites dynamiques — include /etc/nginx/sites-enabled/*.conf
  • Serveur par défaut — Port 443 default_server, proxyfie vers Flask (port 5000)
  • Logs — Format main avec X-Forwarded-For

D. Sécurité

  • HTTPS — Toutes les communications sont chiffrées via TLS
  • Rate limiting — Blocage automatique des IP après 6 tentatives de connexion échouées
  • Authentification — Mot de passe local + LDAP/SSO avec vérification de groupe
  • Sessions Flask — Chiffrées via SECRET_KEY (cookie signé)
  • Fichiers sensibles — Les identifiants OVH sont stockés avec permissions 600
  • Validation Nginx — Chaque modification est validée via nginx -t avant rechargement
  • Rollback automatique — Les configurations invalides sont retirées automatiquement
Recommandations production :
  • Changer SECRET_KEY et ADMIN_PASSWORD avec des valeurs complexes
  • Utiliser des certificats Let's Encrypt plutôt que des auto-signés
  • Restreindre l'accès réseau (firewall, VPN)
  • Sauvegarder régulièrement la configuration (bouton Backup)
  • Configurer LDAP pour une authentification centralisée

E. Dépannage

ProblèmeCause probableSolution
Le conteneur ne démarre pas Port 888 déjà utilisé ou erreur de build docker logs reversproxy puis ss -tlnp | grep 888. Reconstruire avec docker compose up -d --build.
Erreur « 502 Bad Gateway » Service cible inaccessible Vérifier l'adresse et le port cible. Tester : docker exec reversproxy curl -I http://<cible>
Config Nginx ne se recharge pas Erreur de syntaxe dans la configuration docker exec reversproxy nginx -t puis nginx -s reload
LDAP : erreur de connexion Serveur LDAP inaccessible ou identifiants incorrects Utiliser le bouton « Tester la connexion » pour un diagnostic détaillé en 5 étapes
IP bloquée par erreur 6 tentatives de connexion échouées Débloquer depuis le tableau de bord ou attendre 10 minutes. En dernier recours : docker restart reversproxy
Certificat LE échoue Port 80 non accessible ou DNS mal configuré Vérifier que le port 80 est ouvert et que le DNS pointe vers le serveur. Consulter les logs certbot.
Wildcard LE échoue Identifiants OVH incorrects ou propagation DNS Vérifier les clés API OVH. Le challenge DNS-01 attend 60s de propagation (timeout 180s).
Certificat invalide après upload Certificat et clé ne correspondent pas Vérifier : openssl x509 -noout -modulus -in cert.crt | md5sum et openssl rsa -noout -modulus -in cert.key | md5sum (doivent être identiques)
Réinitialisation complète — docker compose down -v && docker compose up -d --build (supprime toutes les données)
